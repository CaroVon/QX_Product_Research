# 产品需求文档 (PRD)：智能产品研究交互工作台

## 1. 产品定位与愿景
**产品名称**：QX Product Research Agent Workspace
**产品愿景**：将传统的“输入主题 -> 黑盒等待 -> 静态输出”的线性产品分析生成工具，升级为具备“断点干预、块级编辑、多轮迭代”能力的现代 SaaS 产品研究工作台。
**目标用户**：资深产品经理 (PM)、用户体验专家 (UX)、工业设计战略家。

## 2. 核心使用流程 (User Journey)
1. **新建项目**：用户输入需要研究的产品名称（如“智能坐姿矫正指环”）。
2. **资料预搜集与审核 (Node 1)**：系统在后台完成 RAG 检索，返回高相关性参考资料列表。用户可人工剔除低质信息，确认后进入下一步。
3. **大纲规划与审核 (Node 2)**：系统生成“产品维度”的 6 大章节大纲。用户在界面上可视化调整大纲节点，确认定稿。
4. **流式生成与渲染 (Node 3)**：系统分章节异步撰写内容，前端打字机式流式（SSE）渲染。
5. **交互式打磨 (Inline AI)**：用户在编辑器中选中某段文本，呼出 AI 悬浮菜单进行局部重写（如“语气更专业”、“补充竞品数据”），通过 Diff 视图确认应用。
6. **最终交付**：一键导出为极具商业质感的 16:9 横版 PPT 风格 PDF。

## 3. 产品最终形态与核心模块 (Core Features)

### 3.1 三栏式沉浸工作台 (Workspace UI)
* **左侧边栏 (Sidebar)**：展示项目列表、当前报告的树状大纲导航。
* **中央编辑器 (Main Editor)**：基于 Tiptap 的块级无头富文本编辑器。支持 Markdown 快捷键，每一段落均为独立的 DocumentBlock，为局部重写提供数据支撑。
* **右侧边栏 (Right Panel)**：多功能交互区。支持“引用溯源（展示 `[^1]` 对应的原文和链接）”和“Agent 对话框（局部修改的指令输入区）”。

### 3.2 节点式状态机 (State Machine)
* 放弃一键到底的执行脚本。
* 后端引入 `PREPARING_DATA`, `WAITING_OUTLINE_APPROVAL`, `DRAFTING`, `COMPLETED` 等状态枚举。
* 前端通过轮询或 WebSocket 监听状态，按需弹出确认卡片，打破黑盒效应。

### 3.3 悬浮交互与溯源闭环 (Interactive Citations)
* 报告正文生成的角标（如 `[^1]`）在编辑器中渲染为可点击的交互组件。
* 点击后，右侧栏平滑滑出，展示对应的参考资料 Metadata（标题、URL、高亮片段），确保研究过程 100% 可溯源。

### 3.4 行间智能体 (Inline AI)
* 用户高亮选中文本，触发 `<BubbleMenu>`（悬浮菜单）。
* 提供“Diff 差异对比视图”，用绿色/红色高亮展示 AI 修改建议，用户点击“应用”后无缝替换当前内容块。


# 开发与实施准则 (Development & Execution Requirements)

## 1. 核心开发原则：优化数据库和完整性
1. 之前的版本中出现太多次数据库问题导致的运行困难，请你想办法解决数据库问题，使得后续运行项目时能快速便捷。
2. 需要在端到端工作流中实现完整任务，无论是否使用Redis（如果使用Redis，必须在Redis中存储项目状态）,我希望看到一个完整项目而不是测试版本。

## 2. 节点推进与积极测试 (Active Testing & Step-by-Step Execution)
为防止架构跑偏，AI Agent 在执行开发任务时，必须严格遵守**“单步验收制”**：
1. **API 优先测试**：在开发前端界面之前，后端每完成一个新节点（如 `/approve-outline`），必须首先编写单元测试，或使用 cURL/Postman 完成接口调通，确认状态机能正确流转至下一步。
2. **阻断式执行**：严禁在“大纲确认接口”未测试通过的情况下，跨步去开发“流式正文生成”模块。**上一步骤 100% 跑通后，才能进入下一步。**
3. **git 仓库管理**：每一小节任务完成后就上传到git仓库，确保代码版本控制。

## 3. 状态管理与容错机制 (Error Handling & Fallback)
* **网络抖动处理**：后端的 Celery 任务在调用大模型或生图 API 时，必须封装完整的 `try-catch` 和 Retry（重试）机制，避免单次 API 超时导致整个项目状态死锁。
* **前端状态锁**：在前端 React Query 轮询过程中，如果当前状态处于 `WAITING_OUTLINE_APPROVAL`（等待用户操作），必须主动暂停后台轮询，防止状态被覆盖或产生多余的网络请求。
* **Diff 撤销机制**：在 Inline AI 局部重写功能中，必须为用户提供明确的 `Discard (放弃修改)` 按钮，确保文档数据的安全性。

## 4. UI/UX 审美与交付红线
* 前端必须杜绝原生的、未经修饰的 HTML 标签，强制使用 Tailwind CSS 进行原子化排版。
* 视觉色彩控制在商业企业级（克制的灰/白底色，低饱和度的品牌主色点缀）。
* 最终导出的 PDF 必须符合 PRD 要求的“16:9 横版 PPT”样式，确保字体大小、留白、强制分页符合商业汇报标准。
